# 🎨 กฎการออกแบบและพัฒนานวนิยายเรื่องใหม่ในอนาคต (Future Story Design Rules)

> [!NOTE]
> เอกสารนี้คือคู่มือเกณฑ์การตัดสินใจออกแบบ (Design Authority Guidelines) เมื่อผู้เขียนหรือนักพัฒนาต้องการเพิ่มนวนิยายเรื่องใหม่เข้ามาในเว็บไซต์ เพื่อคงความสมดุลระหว่าง **"ความกลมกลืนของระบบการอ่านส่วนกลาง"** และ **"ความโดดเด่นมีเอกลักษณ์เฉพาะตัวของวรรณกรรมแต่ละเรื่อง"**

---

## 🏛️ ปรัชญาการออกแบบหลัก (Aesthetic Philosophy)

นวนิยายแต่ละเรื่องในเว็บควรได้รับการออกแบบให้เป็น **"ผลงานศิลปะที่มีชีวิตของตัวเอง"** ผ่านบรรยากาศ แอนิเมชัน และโทนสีที่ไม่ซ้ำกัน:
* **Hvitveldt Lodge**: โทนสีมืด ทารุณ หนาวเหน็บ มีพายุหิมะพัดผ่าน มีเอฟเฟกต์เครื่องพิมพ์ดีดลึกลับสั่นประสาท
* **Tsukinomi Station**: โทนสีอบอุ่นอมเหงา (ส้มฤดูใบไม้ร่วง/เทาฤดูหนาว) มีกลีบซากุระร่วงโรย มีตัวอักษรละลายออกทางสายหมอกแบบ Blur-to-Opaque
* **นิยายเรื่องใหม่**: ควรหาแนวทางภาพ คีย์สี (Accent Color) ฟอนต์ และเอฟเฟกต์แวดล้อมที่เฉพาะตัว (เช่น คลื่นน้ำไหว, ฟองเบียร์ลอย, หรือการวูบวาบของเปลวไฟ) เพื่อสร้างมู้ดและโทนของตัวเองตั้งแต่หน้าแรก

## 🛠️ ระบบการทำงานส่วนกลางที่ทุกเรื่องต้องมีเหมือนกัน (Standardized Shared Core Systems)

เพื่อไม่ให้ผู้อ่านสับสนในส่วนของพฤติกรรมการใช้งานอินเทอร์เฟซ (User Experience) และระบบบันทึกความคืบหน้า ระบบเหล่านี้นักพัฒนา **จำเป็นต้องลอกเลียนแบบหรือดึงโมดูลการทำงานเดียวกันมาใช้**:

### 1. ระบบบันทึกความคืบหน้าการอ่าน (Reading Progress & LocalStorage)
* **การเก็บคีย์ข้อมูล**: ต้องบันทึกสถานะการอ่านบทต่างๆ ลงใน `localStorage` โดยกำหนด Namespace ตามสลักของนิยายแต่ละเรื่อง (เช่น `[novel-slug]:secret:reads` เป็นอาร์เรย์ของไอดีบทที่อ่านจบแล้ว)
* **เกณฑ์ปลดล็อกหน้าของแถม (Extra Page)**:
  * หน้าส่วนเสริมพิเศษ (Extra) จะต้องล็อกไว้เสมอจนกว่าผู้อ่านจะสะสมความคืบหน้าได้อย่างน้อย **80%** ของบททั้งหมด (เช่น Lodge คือ 15/18 บท, Tsukinomi คือ 4/5 ภาค)
  * **ข้อยกเว้นสำคัญ**: ไฟล์ **บทนำ (Chapter 0 / Section 0)** ต้องบันทึกเข้าประวัติว่าอ่านแล้วเมื่อผู้ใช้เข้าชม แต่ **ห้ามนำมาคำนวณร่วมในตัวหารยอดรวมของความคืบหน้า 80%** เพื่อป้องกันปัญหาตัวหารเป็นเศษส่วนหรือล็อกค่านั้นไว้ไม่ให้อ่านข้ามได้
* **ระบบ Bypass**: ต้องคงฟอร์มกรอกรหัสผ่านเพื่อปลดล็อกทันทีสำหรับนักพัฒนา โดยใช้คีย์ตรวจสอบสถานะเป็น `[novel-slug]:extra-bypassed === "true"`

### 2. ระบบนำทางและสารบัญ (Navigation & Table of Contents)
* **หน้าแสดงสารบัญ**: ต้องจัดเรียงบทนำให้อยู่ในลำดับแรกสุดเสมอ (แยกหมวดหมู่เป็น Act หรือ Section ชัดเจน)
* **ป้ายแสดงตำแหน่งล่าสุด (Glow Indicator)**: เมื่อผู้ใช้อ่านค้างไว้ล่าสุดและกลับมาที่สารบัญ ให้มีระบบเพิ่มคลาส CSS เพื่อสั่ง **เรืองแสงสีประจำตัวของนิยายเป็นเวลา 3 วินาที** ตรงลิงก์ของบทที่อ่านล่าสุด เพื่อนำทางผู้อ่านเข้าเนื้อเรื่องได้รวดเร็ว
* **ปุ่มถอยหลัง/ไปต่อ (Prev/Next Chapter Navigation)**:
  * ในหน้ารายละเอียดบทอ่าน ปุ่มนำทางตอนท้ายต้องคำนวณและแสดงลิงก์ถอยหลัง/ไปต่อได้ถูกต้อง
  * ต้องออกแบบการแปลงค่าตัวเลขบทนำ (ตอนที่ 0) ให้ปรากฏเป็นคำว่า **"บทนำ"** เสมอแทนการใช้ตัวเลข 0 หรือ ภาค 0 เพื่อความสวยงาม

### 3. ระบบควบคุมเสียงบรรยากาศ (Audio & Ambient Control)
* **ปุ่มเปิด/ปิดเสียง (Mute Toggle)**: ทุกเรื่องจะต้องติดตั้งคอมโพเนนต์ `<ReaderAmbience />` หรือโมดูลที่คล้ายกันไว้ในมุมใดมุมหนึ่งของหน้าจออ่านหนังสือเสมอ
* **การจำสถานะเสียง**: ต้องเก็บตัวแปรการปิดเสียงชั่วคราว (เช่น `hvitveldt-ambience-muted === "true"`) เพื่อให้เวลาผู้ใช้อ่านตอนถัดไป เพลงจะไม่เด้งกลับมาเปิดเสียงเองจนรบกวนสมาธิผู้ใช้
* **การตั้งค่า Cues**: ทุกบทต้องมีการผูกข้อมูล ID เพลงประกอบเฉพาะตอน (Audio Cues) เพื่อสร้างบรรยากาศขึ้นมาสอดรับกับข้อความในบทนั้นๆ

### 4. ระบบคำโปรยนำบทปูเนื้อเรื่อง (Hook Epigraph Cutscenes)
* **หน้ากากทับจอ (Full-screen Overlay)**: เมื่อผู้อ่านเปิดเข้าบทที่กะไว้เป็นรอยต่อของเรื่อง หน้าจอต้องแสดงผลเป็นแผ่นหลังสีดำทับหน้ากระดาษอ่านเพื่อสร้างอารมณ์ตัดขาด
* **เอฟเฟกต์โปรยปราย**: ต้องดึงเอฟเฟกต์ประจำเรื่อง (เช่น หิมะ หรือ กลีบซากุระ) ขึ้นมาลอยเป็นเลเยอร์พิเศษเหนือก้อนดำนี้ แต่อยู่ใต้ข้อความบรรยาย
* **แอนิเมชันข้อความ**: ข้อความ Hook จะต้องพิมพ์หรือเฟดขึ้นมาเป็นลำดับตามเวลา ไม่เด้งขึ้นมาพร้อมกันทั้งหมด เพื่อบีบมู้ดให้ผู้อ่านค่อยๆ ซึมซับ
* **ปุ่มเข้าเรื่อง**: หลังแอนิเมชันข้อความเล่นจบ ปุ่มสำหรับคลิกเพื่อถอยหน้ากากดำออกและปลดล็อกการเลื่อนจออ่านต้องแสดงผลขึ้นมาอย่างนุ่มนวล

## 🔄 การรองรับเทคโนโลยีพื้นฐาน (Technical & Architecture Rules)

1. **Astro View Transitions (`astro:page-load`)**:
   * โค้ด JavaScript เกือบทั้งหมดต้องผูกการทำงานไว้ใน Event `astro:page-load` เสมอ เพื่อป้องกันไม่ให้สคริปต์หยุดทำงานเมื่อผู้อ่านสับเปลี่ยนหน้าเว็บด้วยกลไก SPA-like Transitions ของ Astro
2. **Mobile Responsiveness**:
   * การจัดวางข้อความ ขนาดฟอนต์ หน้ากากทับจอ ต้องถูกย่อสเกลด้วยหน่วยแบบ Dynamic (เช่น `clamp()`, `vh`, `vw`) เพื่อให้อินเทอร์เฟซมีความเสถียรบนจอโทรศัพท์มือถือและแท็บเล็ตเสมอ
3. **การออกแบบแยกโมดูล (Folder Isolation)**:
   * หลีกเลี่ยงการนำไฟล์สไตล์หรือคอมโพเนนต์ที่ไม่เกี่ยวข้องกันมาปนกันในโฟลเดอร์ของเรื่องอื่น เพื่อช่วยให้นักพัฒนาที่เข้ามาทำโปรเจกต์นี้ในอนาคต สามารถพัฒนาเรื่องที่สาม เรื่องที่สี่ ได้ง่ายโดยไม่กระแทกโมดูลเดิมพังเสียหาย
